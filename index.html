Pipeline de Analytics Logístico

Pipeline end-to-end de análisis de operaciones logísticas: generación de datos sintéticos, limpieza y ETL con Python, análisis SQL con DuckDB y dashboard ejecutivo en Power BI. Incluye recomendaciones ejecutivas y plan de acción basado en los datos.

Stack Python · SQL · DuckDB · Power BI · DAX · Excel
Ver repositorio → Ver dashboard PDF
Área Analytics Logístico
Dataset 5.000 pedidos · 2023–2024
Repositorio GitHub →
Problema

¿Qué problema resuelve?

Las operaciones logísticas generan datos de entregas que rara vez se analizan de forma estructurada. Los equipos no saben qué carriers fallan más, qué zonas tienen peores tiempos ni cuándo ocurren los picos de devoluciones.

Este proyecto construye desde cero una capa de analytics completa que responde esas preguntas con datos, y cierra con recomendaciones concretas para mejorar el OTD global del 61.3% al 75%.

Recomendaciones
SQL · DuckDB

Análisis de performance por carrier y zona

-- OTD y demora promedio por carrier
SELECT
  carrier,
  COUNT(*)                                                         AS total_pedidos,
  ROUND(AVG(CASE WHEN entrega_a_tiempo THEN 1.0 ELSE 0 END)*100, 1) AS otd_pct,
  ROUND(AVG(demora_dias), 2)                                          AS demora_promedio
FROM 'data/processed/fact_orders.csv'
GROUP BY carrier
ORDER BY otd_pct DESC;
Hallazgos

Lo que los datos revelaron

OTD Global · 61.3%

24 puntos por debajo del estándar de industria (85%). 4 de cada 10 pedidos llegan fuera de fecha.

Brecha geográfica crítica

CABA lidera con 74.8% OTD vs. Córdoba en 41.1% — 34 puntos de diferencia entre zonas.

Carrier ganador y a reemplazar

Andreani lidera con 63.9% OTD. Correo Argentino tiene el peor OTD (59.6%) y el mayor costo ($1.354).

Combinación óptima

Rappi Envíos en CABA alcanza 77.6% OTD — los carriers urbanos rinden mejor en proximidad.

Recomendaciones ejecutivas
  • Reasignar volumen de Correo Argentino a Andreani → +2 a +3 pts OTD
  • Implementar regla de carrier por zona (Rappi→CABA, OCA→GBA Norte) → +5 a +8 pts OTD
  • Intervención urgente en Córdoba: ampliar plazo prometido a 5 días → +2 pts OTD
  • Refuerzo operativo en Febrero y Noviembre (meses críticos con OTD < 58%)
  • Potencial total: pasar de 61.3% a ~75% OTD sin inversión tecnológica nueva
Próximas iteraciones
  • Simulación de impacto económico: traducir mejoras de OTD a pesos
  • Score de riesgo por pedido: modelo predictivo de probabilidad de demora
  • Automatización del pipeline con scheduling diario
  • Conexión a datos reales vía API o base de datos PostgreSQL

Automatización de Reportes con Python + n8n

Sistema que elimina trabajo manual en la generación y distribución de reportes operativos diarios.

Stack Python · n8n · SQL
Área Automatización
Estado En desarrollo
Problema

El trabajo repetitivo consume tiempo real

En muchas operaciones, alguien dedica 30-60 minutos por día a copiar datos de un sistema, pegarlos en Excel, formatear un reporte y enviarlo por email. Es trabajo de bajo valor que puede automatizarse completamente.

Este proyecto construye ese flujo automatizado: extracción, transformación, reporte y distribución — sin intervención manual.

Mapa y tabla comparativa de performance geográfica.
